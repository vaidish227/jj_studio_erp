## 2. Tasks Performed (Step-by-Step)

### Task 1 — Diagnose WhatsApp Delivery Failure
- **What:** Designer "Abhishek" was not receiving WhatsApp messages despite mail notifications working fine.
- **Why:** Needed to trace the full pipeline — frontend → backend → Maytapi API.
- **Finding 1:** Notifications were fire-and-forget; errors were silently swallowed with `console.error`.
- **Finding 2:** Maytapi returns HTTP 200 even on failure — the `success: false` field in the body was never checked.
- **Finding 3:** The phone number `9617980134` (10 digits, no country code) was being sent to Maytapi, which accepted it but formed chatId `9617980134@c.us` — a non-existent/wrong WhatsApp JID. The correct JID for India should be `919617980134@c.us`.

### Task 2 — Fix Phone Number Normalization
- Added `normalizePhone()` to the Maytapi provider to strip spaces, dashes, and parentheses.
- Added a minimum-digit check: if the cleaned number has fewer than 11 digits, it throws with a clear error message directing the user to use international format (`+91 XXXXXXXXXX`).
- Added the same normalization + validation in `updateUserContact` controller so bad numbers are rejected at save time.

### Task 3 — Make Notification Dispatch Synchronous & Return Results
- Changed `dispatchTaskNotifications` from fire-and-forget (`.then().catch()`) to a proper `async` function that returns a `results` object `{ mail: { sent, reason }, whatsapp: { sent, reason } }`.
- Changed `createTask` to `await` the notification dispatch and include `notificationResults` in the 201 response.
- Updated `useTaskForm.js` to read `res.notificationResults` and show amber toast warnings for skipped or failed notifications with human-readable messages.

### Task 4 — Fix Maytapi Success Check
- Added check for `response.data?.success === false` in the Maytapi provider.
- Previously, a Maytapi API rejection (HTTP 200 but `success: false`) was treated as a successful send.
- Now it throws with Maytapi's own error message, which surfaces in the UI toast.
- Added `console.log("[maytapi] response:", ...)` for server-side debugging.

### Task 5 — Add WhatsApp Number to User Creation Form

### Task 6 — Fix Role Dropdown Clipping (Settings Page)

## 8. State Management / Logic

### `dispatchTaskNotifications` (Task.controller.js)
```
Input:  { task, project, assignedUser, actorId, notifyMail, notifyWhatsApp }
Output: { mail: { sent: bool, reason?: string },
          whatsapp: { sent: bool, reason?: string } }

Logic:
  if notifyMail:
    if !assignedUser.email → { sent: false, reason: "no_email" }
    else → try mailService.sendImmediate() → { sent: true } or { sent: false, reason: error.message }

  if notifyWhatsApp:
    if !assignedUser.phone → { sent: false, reason: "no_phone" }
    else → try whatsappService.sendImmediate() → { sent: true } or { sent: false, reason: error.message }
```

### `normalizePhone` (maytapi.provider.js)
```
Input:  raw phone string e.g. "+91 9876 543 210" or "9876543210"
Steps:
  1. Strip all spaces, dashes, dots, parentheses
  2. If doesn't start with "+", prepend "+"
  3. Count digits — if < 11, throw Error("missing country code...")
Output: "+919876543210"
```

### Team Modal State (`ManageTeamModal.jsx`)
```
Local state:
  draft: { primaryDesigner: User|null, designerB: User|null, ... }
  saving: boolean

On open (useEffect on isOpen):
  Seeds draft from project prop (populated team objects)

On EmployeePicker change:
  setDraft(prev => ({ ...prev, [field]: user }))

On save:
  Builds payload: { [field]: draft[field]?._id || '' }
  Calls pmsService.updateTeam(project._id, payload)
  On success: toast.success, onSaved(), onClose()
```

### `canManage` check (`TeamTab.jsx`)
```js
const { user } = useAuth();
const canManage = ['admin', 'md', 'manager'].includes(user?.role);
```
- Role comes from JWT decoded in `AuthContext` and stored in localStorage
- No async permission lookup needed — deterministic from role string

## 9. Challenges Faced

### Challenge 1: WhatsApp Sends "Success" But Message Never Arrives
- **Root Cause:** Maytapi accepts any phone number and returns HTTP 200 + `success: true` even when it forms an invalid WhatsApp JID. `9617980134` (10 digits) → chatId `9617980134@c.us` → no real WhatsApp account.
- **Resolution:** Added minimum 11-digit check in `normalizePhone`. Throws with actionable message directing user to use `+91 XXXXXXXXXX` format. DB update also validates before saving.

### Challenge 2: Errors Were Invisible to the User
- **Root Cause:** All notification dispatch was fire-and-forget. Errors only appeared in server console logs which users never see.
- **Resolution:** Made dispatch synchronous; returned results in API response; frontend reads and shows amber toasts.

### Challenge 3: Role Dropdown Clipped in Settings Page
- **Root Cause:** The accordion container had `overflow-hidden` to clip child border-radius. This also clipped the absolutely-positioned dropdown from the `Select` component.
- **Resolution:** Removed `overflow-hidden`; applied conditional `rounded-t-2xl` / `rounded-2xl` directly to the toggle button so visual rounding is preserved without clipping dropdowns.

### Challenge 4: Team Permission Tied to DB-Stored Permissions
- **Root Cause:** `requirePermission("projects.update")` queries the `Role` MongoDB collection. If the role document isn't seeded with the right permissions, MD/Manager can't manage teams even though they should always be able to.
- **Resolution:** Switched to `requireRole("admin", "md", "manager")` — hardcoded in code, independent of DB state.

### Challenge 5: Phone Format Appearing Correct But Wrong for Maytapi
- **Root Cause:** The previous `normalizePhone` added `+` prefix to any number — turning `9617980134` into `+9617980134` (Lebanon prefix `+961`). Maytapi sent to this invalid JID.
- **Resolution:** Length check rejects numbers with < 11 digits after normalization, forcing proper country code entry.

## 12. Notes for Future Reference

### Maytapi Behaviour — Critical
- HTTP 200 does **not** mean the message was delivered. Always check `response.data.success`.
- ChatId format for WhatsApp: `{countryCode}{number}@c.us` — e.g. `919876543210@c.us` for India.
- Phone numbers **must** be stored in international format: `+91XXXXXXXXXX`. A bare 10-digit number will be sent to the wrong or non-existent JID without any error.
- Maytapi group creation API: `POST /api/{productId}/{phoneId}/createGroup` — supported and can be integrated.

### Auth Middleware — Two Guards Available
```js
requirePermission("projects.update")  // DB-driven; checks Role collection
requireRole("admin", "md", "manager") // Code-driven; checks JWT role field
```
Use `requireRole` for features where access should never depend on DB configuration. Use `requirePermission` for features that admins should be able to configure at runtime.

### `Select` Component — onChange Contract
The shared `Select` component calls `onChange(value)` with the raw string value, **not** a synthetic event. Always use:
```jsx
// ✅ Correct
onChange={(value) => setField('taskType', value)}
// ❌ Wrong — e is already the string, e.target is undefined
onChange={(e) => setField('taskType', e.target.value)}
```

### EmployeePicker — Returns Full User Object
`EmployeePicker.onChange` receives the full user object `{ _id, name, email, role, activeTasks }` or `null` on clear. To get just the ID for API calls: `user?._id || ''`.

### SLOTS Export Pattern
`ManageTeamModal.jsx` exports both the component (default) and the `SLOTS` config array (named). This pattern avoids duplicating the slot definitions across files:
```js
import ManageTeamModal, { SLOTS } from '../ManageTeamModal';
```

### Overflow & Dropdowns
Any component that renders an absolutely-positioned dropdown (Select, EmployeePicker, DatePicker) must **not** be placed inside a container with `overflow: hidden`. Use `overflow: visible` (default) and handle rounded corners per-child instead.
