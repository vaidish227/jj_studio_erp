## 9. Challenges Faced

### Challenge 1 — Route Shadowing Risk (DLR Endpoint)
**Problem**: In Express, `/project/:projectId` would match `/dlr/:projectId` if registered first, since both have one param segment after a literal.

**Resolution**: Registered `router.get("/dlr/:projectId", ...)` **before** `router.get("/project/:projectId", ...)` in `Drawing.route.js`. This is consistent with how other static-before-parameterised patterns are handled throughout the codebase.

### Challenge 2 — Wrong API Method (ApprovalDashboardPage)
**Problem**: `pmsService.getPendingApprovals()` is the drawings pending-approval endpoint (no user context, returns `{ drawings: [] }`), but `ApprovalDashboardPage` was calling it with `user._id` and expecting `{ approvals: [] }`. Both methods exist in pmsService — they just serve different purposes.

**Root Cause**: The Approval model (for formal requests: quotations, materials, etc.) and the Drawing model (for design approvals) are separate. The page is for the `Approval` model.

**Resolution**: Changed to `getPendingApprovalsByUser(user._id)` which maps to `GET /api/pms/approval/pending/:userId`.

### Challenge 3 — DrawingsTab Design Decision
**Problem**: Should the project-level Drawings tab show a simplified read-only view, or the full interactive DrawingCard?

**Decision**: Full interactive DrawingCard — because PMs and designers both need to act on drawings directly from the project context, not just from the global Drawing Library. The project context (knowing which project you're in) makes it more natural to manage drawings from there.

**Trade-off**: The parent `ProjectDetailPage` now passes `onDrawingUpdated={refreshDrawings}` which re-fetches all project drawings from the API when a drawing is updated. This is an acceptable API call pattern for a feature used intentionally.

## 10. Improvements / Refactoring Done

### DrawingsTab — Added Filtering
The old DrawingsTab had no filters. The rewrite added client-side status and drawing-type filters, matching the experience in `DrawingLibraryPage`. This required no extra API calls since the parent already fetches all drawings.

### getDLRSheet — Lean Query + Group in Node
Rather than multiple aggregation pipeline stages in MongoDB, the DLR handler fetches all drawings with `.lean()` and groups them in Node.js memory. Given typical project sizes (< 500 drawings), this is simpler and equally performant. MongoDB aggregation would be needed only if DLR sheets grow to thousands of drawings.

### Task Status Logic — Reused Existing Endpoint
For `pending_client_approval` and `completed` transitions, the decision was made to reuse the existing `updateTask` PUT endpoint rather than adding dedicated `sendForClientApproval` and `completeTask` endpoints. This works because:
- `Task.validator.js` already includes both statuses in `TASK_STATUSES`
- `Task.controller.updateTask` already auto-sets `completedAt` for terminal statuses
- The frontend enforces business rules via `canSendToClient` / `canMarkCompleted` flags (only PM/PC/MD can see these buttons, and only when the task is in the right status)

## 12. Notes for Future Reference

### Route Ordering in Express
Always register specific static-path routes before parameterised routes. In this codebase, the pattern is consistently:
```
router.get("/static-segment", handler)        // ← first
router.get("/:param",         handler)        // ← after
```
This applies to both backend route files and React Router (`App.jsx`). See the `/tasks/:id` before `/tasks` comment in App.jsx.

### Task Status Transitions — Business Rules
| Transition | Who | Endpoint |
|-----------|-----|----------|
| `not_started → in_progress` | Designer | `PUT /task/update/:id` |
| `in_progress → pending_review` | Designer | `PATCH /task/submit/:id` |
| `revision_requested → in_progress` | Designer | `PUT /task/update/:id` |
| `pending_review → approved` | PM/PC/MD | `PATCH /task/approve/:id` |
| `pending_review → revision_requested` | PM/PC/MD | `PATCH /task/request-revision/:id` |
| `approved → pending_client_approval` | PM/PC/MD | `PUT /task/update/:id` |
| `pending_client_approval → completed` | PM/PC/MD | `PUT /task/update/:id` |
| Any status → `released_to_site` | System (auto) | Triggered by `Drawing.controller.releaseDrawing` |
| Any → Any | PM/PC/MD | `PUT /task/update/:id` (bypass, allowed by updateTaskSchema) |

### Drawing Status Transitions — Business Rules
| Transition | Who | Endpoint |
|-----------|-----|----------|
| (new) → `draft` | Designer | `POST /drawing/upload` |
| `draft/rejected → sent_for_approval` | Designer | `PATCH /drawing/send-for-approval/:id` |
| `sent_for_approval → approved` | PM/PC/MD | `PATCH /drawing/approve/:id` |
| `sent_for_approval → rejected` | PM/PC/MD | `PATCH /drawing/reject/:id` |
| `approved → released_to_site` | PM/PC/MD | `PATCH /drawing/release/:id` |
| (any) → `draft` (new version) | Designer | `POST /drawing/revise/:id` (archives old version) |

### Auto-Propagation Logic (Drawing ↔ Task)
- **Task submitted**: Linked drawings `draft/rejected → sent_for_approval` (batch update)
- **Task approved**: Linked drawings `sent_for_approval → approved` (batch update)
- **Task revision requested**: Linked drawings `sent_for_approval → rejected` (batch update)
- **Drawing released to site**: Linked task → `released_to_site` (single update in `releaseDrawing`)

### Designer Visibility Isolation
- Designers only see projects they are assigned to (via `useProjects.js` → `getMyProjects` when `user.role === 'designer'`)
- `getMyProjects` filters: `{ $or: [primaryDesigner, designerB, designerC, designerD, designerE, supervisor, contractor] === userId }`
- All other roles see all projects via `getAllProjects`

| `tasks.approve` | manager, md |

### Critical Post-Deployment Step
```bash

```
This must be run once after deployment or when new team members are added. It syncs the `Role` model in MongoDB with the permissions defined in the seed script. Without it, new permissions (`tasks.submit`, `tasks.approve`, `designer.dashboard`, etc.) won't be assigned to existing roles.
