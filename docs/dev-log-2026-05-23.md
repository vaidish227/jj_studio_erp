## 2. Tasks Performed (Step-by-Step)

### Task 1 — Fix Custom `Select` Component `onChange` Mismatch
**Time:** Morning  
**What:** The shared `Select` component calls `onChange(value)` directly (passing the raw string), but multiple consumer components were treating it like a native DOM event using `e.target.value`. This caused silent failures — selecting an option appeared to work visually but no state was updated.  
**Why:** The `Select` is a custom component, not a native `<select>`, so it does not produce an event object.  
**Files Fixed (5 total):**
- `CreateVendorModal.jsx` — Category select
- `UploadDrawingModal.jsx` — Drawing Type select
- `DrawingsTab.jsx` — Type filter select
- `DrawingLibraryPage.jsx` — Drawing type filter
- `CreateProjectModal.jsx` — Project Type select

**Change Pattern:**
```jsx
// Before (broken)
onChange={(e) => set('category', e.target.value)}

// After (correct)
onChange={(val) => set('category', val)}
```

```jsx

```

### Task 4 — Fix Infinite Loader on Client Approvals Tab
**Time:** Late Morning  
**What:** The "Client Approvals" tab on the Approval/Review Design page showed a loading spinner that never resolved.  
**Root Cause (2 bugs):**

**Bug 1 — Frontend (`ReviewDesignPage.jsx`):**
The `fetchApprovals` function had an early return guard:
```js
if (!user?._id) return;  // ← never called setIsLoading(false)
```
Since `isLoading` initialised as `true`, if `user._id` wasn't ready when the component mounted, the spinner stayed forever.

**Fix:**
```js
if (!user?._id) { setIsLoading(false); return; }
```

**Bug 2 — Backend (`Approval.controller.js`):**
`getPendingApprovals` hardcoded `status: "pending"` in the query, so the frontend's Approved/Rejected/All filter tabs were always empty. Also `requestedBy` was not populated.

**Fix:**
```js
const query = { approverId: req.params.userId };
if (req.query.status) query.status = req.query.status;

const approvals = await Approval.find(query)
  .populate("projectId",   "name trackingId")
  .populate("requestedBy", "name")
  .sort({ createdAt: -1 });
```

### Task 6 — Fix Horizontal Scrollbar in Collapsed Sidebar
**Time:** Afternoon  
**What:** A horizontal scrollbar appeared at the bottom of the icon-only sidebar.  
**Root Cause:** The nav element had `overflow-y: auto`, which per the CSS spec implicitly sets `overflow-x: auto` as well. The active-state left-bar (`-left-[15px]`) and tooltip (`left-full`) exceeded the nav's horizontal bounds, triggering the scroll.  
**Fix:** Added `overflow-x-hidden` to the nav in `Sidebar.jsx`.

### Task 7 — Add Hover Flyout Menus in Collapsed Sidebar
**Time:** Afternoon  
**What:** When hovering over any icon in collapsed mode, a flyout panel appears to the right showing all navigable options. Users can click any item to navigate.  
**Why:** In icon-only mode, users have no visible labels — the flyout bridges discoverability.

**For Groups (CRM, KIT, Projects, etc.):**
- Flyout shows the group name as a gold uppercase header
- All child items listed as clickable rows
- Clicking any child navigates and closes the flyout

**For Standalone Items (Dashboard, Clients, Reports):**
- Flyout shows a small card with icon + label, clickable to navigate

**Hover Bridge Pattern (prevents flyout closing mid-travel):**
```
onMouseEnter icon  → openFlyout() immediately, cancel any pending close
onMouseLeave icon  → schedule close after 150ms
onMouseEnter panel → cancel close timer (stays open)
onMouseLeave panel → schedule close again
```

**Why `position: fixed` + `ReactDOM.createPortal`:**
The flyout is rendered via `createPortal` into `document.body`. This escapes:
- The sidebar's `overflow-hidden` which would clip absolute children
- The sidebar's stacking context which placed it behind main content

### Task 8 — Fix Flyout Appearing Behind Main Content
**Time:** Late Afternoon  
**What:** Flyout menus were rendering beneath the dashboard content even with `z-[9999]`.  
**Root Cause:** The sidebar (being earlier in the DOM) has a lower base stacking context than the content area. Even high z-index values on children of the sidebar can't override elements in the content area's stacking context.  
**Fix:** `ReactDOM.createPortal(panel, document.body)` renders the flyout as a direct child of `<body>` — a sibling to everything, not a descendant of the sidebar. Combined with `z-[99999]`, this guarantees the flyout always renders on top.

## 8. State Management / Logic

### Sidebar Collapse State
```js
// AppLayout.jsx
const [isCollapsed, setIsCollapsed] = useState(
  () => localStorage.getItem('sidebar-collapsed') === 'true'
);

const toggleCollapsed = () =>
  setIsCollapsed((prev) => {
    localStorage.setItem('sidebar-collapsed', String(!prev));
    return !prev;
  });
```
- **Scope:** Local to `AppLayout`, passed down as props
- **Persistence:** `localStorage` key `'sidebar-collapsed'`
- **Mobile:** Ignored — mobile always uses full-width overlay drawer

### Flyout Hover Timer Pattern
```js
// SidebarGroup.jsx & SidebarItem.jsx
const closeTimer = useRef(null);

const flyoutControls = {
  cancel:   () => clearTimeout(closeTimer.current),
  schedule: () => { closeTimer.current = setTimeout(() => setFlyoutOpen(false), 150); },
  now:      () => { clearTimeout(closeTimer.current); setFlyoutOpen(false); },
};
```
- **Purpose:** Gives the mouse 150ms to travel from icon to flyout without triggering a close
- **Pattern:** Same debounced-close pattern used in `ProfileDropdown`'s Settings submenu

---

### Portal Position Calculation
```js
const openFlyout = () => {
  clearTimeout(closeTimer.current);
  if (iconRef.current) {
    setFlyoutTop(iconRef.current.getBoundingClientRect().top);
  }
  setFlyoutOpen(true);
};

// Viewport clamping prevents flyout going below screen
const clampedTop = Math.min(top, window.innerHeight - (children.length * 40 + 60));
```

## 12. Notes for Future Reference

### Rule: Custom `Select` always receives raw value, not an event
The shared `Select` component at `frontend/src/shared/components/Select/Select.jsx` calls `onChange(opt.value)` — a plain string, not a DOM event. Any component using `<Select>` must handle this:
```jsx
// Correct
<Select onChange={(val) => setState(val)} />

// Wrong — will break silently
<Select onChange={(e) => setState(e.target.value)} />
```

---

### Rule: `EmployeePicker` returns the full user object
`EmployeePicker` passes `{ _id, name, role, email, activeTasks }` to `onChange`. When sending to an API that expects an ObjectId string, always extract `._id`:
```jsx
const payload = { assignedTo: selectedUser._id };
```

---

### Rule: All floating UI in the sidebar must use `ReactDOM.createPortal`
The sidebar is earlier in the DOM than the main content. Any `position: absolute/fixed` element with `z-index` inside the sidebar will still lose to the content area's natural stacking order unless rendered via portal into `document.body`.

---

### Rule: Setting `overflow-y: auto` implicitly enables `overflow-x: auto`
Per the CSS specification, you cannot have `overflow-y: auto` and `overflow-x: visible` simultaneously. The browser silently upgrades `overflow-x` to `auto`. To prevent unintended horizontal scrollbars, always pair `overflow-y-auto` with `overflow-x-hidden`.

---

### `ProfileDropdown` outside-click with portals
When the dropdown panel is rendered via portal (outside the `wrapperRef` subtree), `useClickOutside` alone is not sufficient. Use a separate `document.mousedown` listener that checks both refs:
```js
const handler = (e) => {
  const inWrapper = wrapperRef.current?.contains(e.target);
  const inPanel   = panelRef.current?.contains(e.target);
  if (!inWrapper && !inPanel) setIsOpen(false);
};
```
