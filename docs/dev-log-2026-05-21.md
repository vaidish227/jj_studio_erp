## 2. Tasks Performed (Step-by-Step, Chronological)

### Step 1 — Critical Security Fixes (Backend)

**Task 1.1 — `auth.service.js`: Block inactive users at login**
- **What:** Added an `isActive` check immediately after finding the user in the database, before password comparison.
- **Why:** Inactive users (marked `isActive: false` by admin) could still log in and receive a valid JWT. The deactivation had no effect at the auth layer.

**Task 1.2 — `auth.middleware.js`: Fix `customPermissions` never being applied**
- **What:** Replaced `req.user?.customPermissions || []` with a live DB lookup using `User.findById(...).select('customPermissions isActive')`.
- **Why:** The JWT token only stores `{id, email, role}`. Custom permissions are never embedded in the token. The old code always returned an empty array for `customPermissions`, making per-user overrides completely non-functional at the backend.
- **Also added:** An `isActive` check inside `loadPermissions` — if a user is deactivated mid-session (token still valid), subsequent requests are now blocked with `403 Account is inactive`.

**Task 1.3 — `Roles.controller.js`: Validate `customPermissions` on update + add `getEffectivePermissions` endpoint**
- **What (Validation):** Before saving `customPermissions` to a user, every permission string is validated against `ALL_PERMISSIONS`. Returns `400` with a list of invalid strings if any fail.
- **Why:** Without validation, an admin could accidentally save a typo as a custom permission, creating phantom grants that would never match a real permission check.
- **What (New Endpoint):** Added `getEffectivePermissions(req, res)` which returns:
  ```json
  {
    "userId": "...",
    "role": "designer",
    "rolePermissions": [...],
    "customPermissions": [...],
    "effective": [... merged unique array ...]
  }
  ```
- **Why:** The frontend User Overrides panel needs to display inherited permissions separately from custom overrides — this single endpoint delivers exactly that.

**Task 1.4 — `Roles.route.js`: Wire the new endpoint**
- Added `GET /api/roles/users/:userId/effective-permissions` before the existing `PATCH /:userId` routes to prevent Express route-order ambiguity.

**Task 2.2 — `Role.model.js`: Sync `ALL_PERMISSIONS` with new tab strings**
- Added all 15 tab permission strings to the backend `ALL_PERMISSIONS` array.
- **Why:** `ALL_PERMISSIONS` is the authoritative validation list. Any permission string not in this array is rejected by the updated `updateUser` controller. New tab perms must be registered here before they can be assigned.

**Task 2.3 — `navigation.js`: Explicit permissions on all nav children**
- **Before:** Most nav children had no `permission` field and silently inherited from their parent (e.g., all CRM children were visible to anyone with `crm.read`).
- **After:** Every nav child now has its own explicit permission:
  - `crm-clients` → `crm.tab.clients`
  - `new-leads` → `crm.tab.leads`
  - `assign-task` → `projects.tab.assign` (was `tasks.create` — action gate, not nav gate)
  - `review-design` → `projects.tab.review`
  - `users` → `settings.tab.users`
  - `roles-permissions` → `settings.tab.roles`
- **Why:** Tab permissions decouple "can see this section in the menu" from "can perform actions inside it". A Designer can have `tasks.submit` but not `projects.tab.assign` — they never see the Assign Task page in the nav.

### Key Logic: `togglePermissionSet`

```javascript
// Atomically toggle a list of full permission strings — used by ModuleCard's all-toggle
const togglePermissionSet = (permSet) => {
  setDraftPermissions((prev) => {
    // Read from prev (not outer scope) — prevents stale closure
    const allActive = permSet.every((p) => prev.includes(p));
    const next = allActive
      ? prev.filter((p) => !permSet.includes(p))   // revoke all
      : [...new Set([...prev, ...permSet])];          // grant all (deduped)
    setIsDirty(true);
    return next;
  });
};
```

### Key Logic: `requirePermission` Middleware (after fix)

```javascript
const requirePermission = (permission) => async (req, res, next) => {
  if (!req.permissions) {
    const role = await Role.findOne({ name: req.user?.role }).lean();
    const userDoc = await User.findById(req.user?.id).select('customPermissions').lean();
    req.permissions = [...new Set([
      ...(role?.permissions || []),
      ...(userDoc?.customPermissions || [])
    ])];
  }
  if (!hasPermission(req.permissions, permission)) {
    return res.status(403).json({ message: `Access denied. Requires: ${permission}` });
  }
  next();
};
```

### 3-Level Permission Architecture

```
Level 1 — Module Gate (module access)
    Example: crm.read, projects.read
    Purpose: Controls if a user can access the module at all
    Enforced: Backend routes + frontend PermissionGate component

Level 2 — Tab Gate (sub-section visibility)
    Example: crm.tab.clients, settings.tab.roles
    Purpose: Controls which nav children / page sections are visible
    Enforced: navigation.js permission field → sidebar hides item if missing

Level 3 — Action Gate (specific operations)
    Example: tasks.approve, drawings.release
    Purpose: Controls buttons, forms, and API calls within a page
    Enforced: Backend requirePermission() + frontend PermissionGate
```

## 9. Challenges Faced

### Challenge 1 — Stale Closure in `toggleModule`
**Problem:** The old `toggleModule` read `draftPermissions` from the outer closure of the hook. When called rapidly or in tests, the value could be stale from a previous render.

**Resolution:** Rewrote all state-setting functions to read the current value from the `setState(prev => ...)` callback pattern instead of the outer scope variable. This is the React idiomatic pattern for dependent state updates.

---

### Challenge 2 — `customPermissions` Silently Ignored at Backend
**Problem:** The JWT token only stores `{id, email, role}`. The old middleware did `req.user?.customPermissions || []` which always returned `[]`. Per-user permission overrides were completely non-functional at the API level — only enforced on the frontend via localStorage.

**Resolution:** Added a DB lookup in both `loadPermissions` and `requirePermission` to fetch `customPermissions` and `isActive` from the User document on every authenticated request. The extra DB query is minimal — only `select('customPermissions isActive')` is fetched (no password, no other fields).

---

### Challenge 3 — Navigation Access Regression Risk
**Problem:** Adding explicit tab permissions to `navigation.js` children means existing users who don't have those new tab permissions will suddenly lose access to nav sections they previously had (because they no longer inherit from the parent).

**Resolution:** Updated `seedRoles.js` to include all 15 new tab permissions in the appropriate system roles. Admin has `['*']` so no change needed. An explicit note was added: **"Run `node backend/src/scripts/seedRoles.js` after deployment."**

---

### Challenge 4 — Professional UI for Permission Matrix
**Problem:** The flat module-row list (horizontal rows in a panel) was functional but difficult to scan for admins managing many roles. User feedback: "not looking good and professional."

**Two Redesign Iterations:**
1. First iteration (card grid attempt) — had uneven card heights and poor alignment across columns
2. Second iteration (flat list with fixed-width columns) — clean alignment but still felt like a data table
3. **Third iteration (today — final):** 2-column card grid with colored group headers, accent bars, and self-contained module cards

## 12. Notes for Future Reference

### Critical Rule: Always Read Current State in `setState` Callbacks
```javascript
// ❌ WRONG — reads stale outer-scope value
const toggle = (perm) => {
  const next = draftPermissions.includes(perm) 
    ? draftPermissions.filter(p => p !== perm)
    : [...draftPermissions, perm];
  setDraftPermissions(next);
};

// ✅ CORRECT — reads current state from callback
const toggle = (perm) => {
  setDraftPermissions((prev) => 
    prev.includes(perm) ? prev.filter(p => p !== perm) : [...prev, perm]
  );
};
```

### JWT Does NOT Carry Custom Permissions
The token payload is `{id, email, role}` only. Any backend code that reads `req.user.customPermissions` will always get `undefined`. Always fetch from DB when custom permissions are needed server-side.

**Task 3.1 — `WhatsAppGroup.controller.js`: Full rewrite with member management + sync**

Key additions:

When a group of type `main/drawing/supervision/payment` is created, the corresponding `kickstartData.*GroupCreated` boolean is auto-set to `true` on the Project document.

**`tryProviderOp(fn)` helper:**
A non-throwing wrapper for all Maytapi API calls. Returns `{success, result}` or `{success: false, error}`. Ensures ERP operations always complete even if the WhatsApp provider fails.

- Prevents duplicate phones — returns 409 if `phone` already in group
- Adds member to ERP group first (always succeeds)

- Phone decoded from URL param via `decodeURIComponent(req.params.phone)` (frontend sends URL-encoded phone)

- Pre-check: if `phonesWithNumbers.length === 0` → returns 400 with actionable message

- If `providerGroupId` exists: calls `addGroupMember` for each phone (re-sync mode)
- Sets `syncStatus`: `synced` (0 errors) or `partial` (some errors) or `failed` (createGroup failed)

**`updateGroup`:** Extended to auto-reset `syncStatus → 'unsynced'` and clear `syncErrors` when a new `providerGroupId` is set. This makes "Re-sync Members" button appear immediately after manual group linking.

## 7. APIs / Services / Integrations

### Maytapi External API Calls

| Method | Maytapi Endpoint | ERP Function | Purpose |
|--------|-----------------|--------------|---------|
| POST | `/api/{productId}/{phoneId}/createGroup` | `createGroup` | Creates a new WhatsApp group with initial members |
| POST | `/api/{productId}/{phoneId}/addGroupParticipant` | `addGroupMember` | Adds one participant to an existing group |
| POST | `/api/{productId}/{phoneId}/removeGroupParticipant` | `removeGroupMember` | Removes a participant from a group |
| GET  | `/api/{productId}/{phoneId}/getGroup/{groupId}` | `getGroupInfo` | Fetches current group info (members, admins) |

**Credentials:** Read from `process.env.MAYTAPI_PRODUCT_ID`, `MAYTAPI_PHONE_ID`, `WHATSAPP_TOKEN` (or overridden per-call via `config` param).

**Phone number format:** Maytapi requires digits only — no `+` prefix. `normalizePhone()` strips whitespace/dashes/brackets/parens, ensures leading `+`, validates min 11 digits, then `.replace('+', '')` before sending to Maytapi endpoints.

### New ERP API Endpoints

| Method | Path | Controller | Purpose | Auth |
|--------|------|-----------|---------|------|
| GET | `/api/pms/whatsapp-group/all` | `getAllGroups` | Paginated cross-project group listing | `projects.read` |
| GET | `/api/pms/whatsapp-group/:id` | `getGroupById` | Single group with full population | `projects.read` |
| POST | `/api/pms/whatsapp-group/:id/members` | `addMember` | Add one member to ERP group + optional WA sync | `pms.whatsapp.manage` |
| DELETE | `/api/pms/whatsapp-group/:id/members/:phone` | `removeMember` | Remove member from ERP group + optional WA sync | `pms.whatsapp.manage` |
| POST | `/api/pms/whatsapp-group/:id/sync` | `syncWithProvider` | Create WA group or re-sync members | `pms.whatsapp.manage` |

### Hybrid Architecture

ERP groups always exist in MongoDB (the source of truth). Maytapi sync is optional and additive:

```
syncStatus: 'unsynced'  → Group exists in ERP only (not yet in WhatsApp)
syncStatus: 'synced'    → ERP group has a providerGroupId, all members added to WA
syncStatus: 'partial'   → Sync succeeded but some members failed (e.g. invalid phone)
syncStatus: 'failed'    → Maytapi createGroup itself failed
```

## 8. State Management / Logic

### `useWhatsAppGroups` Hook — State Map

```
State:
├── groups[]       — groups for the current project (or empty if no projectId)
├── isLoading      — set true at effect start, false when done
├── error          — error message string or null
├── version        — integer incremented by refresh() to trigger re-fetch

Operations (all wrapped in useCallback):
├── refresh()              — increments version → triggers effect re-run
├── createGroup(data)      — POST create, then refresh
├── updateGroup(id, data)  — PUT update, then refresh
├── deleteGroup(id)        — DELETE, then refresh
├── addMember(gId, data)   — POST add member, then refresh
├── removeMember(gId, ph)  — DELETE member (phone URL-encoded), then refresh
├── syncGroup(id)          — POST sync, then refresh
├── sendUpdate(id, data)   — POST broadcast, then refresh
```

### Key Logic: `freshManagingGroup` Pattern

```javascript
// Keep ManageGroupMembersModal in sync with latest data after member changes
// without closing the modal.
const freshManagingGroup = managingGroup
  ? (groups.find((g) => g._id === managingGroup._id) || managingGroup)
  : null;
```
When `addMember`/`removeMember` is called, `refresh()` triggers a re-fetch. The `groups` array updates, and `freshManagingGroup` re-derives from the fresh array by ID — the modal sees the updated member list without unmounting.

### Key Logic: `tryProviderOp` Pattern

```javascript
const tryProviderOp = async (fn) => {
  try {
    const result = await fn();
    return { success: true, result };
  } catch (err) {
    console.warn("[WhatsAppGroup] provider op failed:", err.message);
    return { success: false, error: err.message };
  }
};
```
All Maytapi calls go through this wrapper. The controller checks `result.success` and updates `syncStatus`/`syncErrors` accordingly, but never throws — ERP operations always complete.

### Key Logic: Phone URL Encoding

```javascript
// Frontend (pmsService.js)
removeWhatsAppGroupMember: (id, phone) =>
  apiClient.delete(`/pms/whatsapp-group/${id}/members/${encodeURIComponent(phone)}`),

// Backend (controller)
const phone = decodeURIComponent(req.params.phone);
```
Phone numbers like `+919617980134` contain `+` which URL-parsers interpret as a space. Without encoding, the backend receives `919617980134` (no plus) and the member-lookup fails.

### Key Logic: Auto-reset syncStatus on Manual Link

```javascript
// updateGroup controller
if (value.providerGroupId) {
  updateData.syncStatus = "unsynced";
  updateData.syncErrors = [];
}
```
When the user pastes a group ID via `LinkManuallyModal`, `syncStatus` resets to `unsynced` automatically. This causes `GroupCard` to show "Re-sync Members" instead of "Link Manually" — guiding the user to the next step.

## 9. Challenges Faced

### Challenge 1 — Maytapi 502: Empty Numbers Array
**Problem:** On first sync attempt, the group had members but none with phone numbers. The frontend showed `"CreateGroup could not complete."` from a 502 response.

**Investigation:** Traced to `phonesWithNumbers = []` → Maytapi `/createGroup` called with `{ name: "...", numbers: [] }`. Maytapi requires at least 1 participant.

**Resolution:** Two-layer fix:
1. Backend pre-check returns 400 with actionable message before calling Maytapi
2. Provider-level guard throws a descriptive error if `normalizedParticipants.length === 0`

---

### Challenge 2 — Maytapi 502 with Valid Phone Numbers
**Problem:** After adding 2 members with valid Indian phone numbers (`91 9617980134`, `+917773091428`), sync still returned `"CreateGroup could not complete."` 502.

**Investigation:** Pre-check was passing (phones existed). Numbers were normalizing correctly (`919617980134`, `917773091428`). Conclusion: Maytapi server-side rejection — not an ERP code issue. Possible causes: phone session disconnected, plan doesn't support group creation, or phone numbers aren't active WhatsApp users.

**Resolution:** The ERP cannot fix Maytapi's server state. Built a full UX fallback:
- Improved error display on GroupCard explaining the likely causes
- `LinkManuallyModal` with troubleshooting checklist and group ID input
- Backend `updateGroup` auto-resets sync state when a new group ID is saved
- After linking, "Re-sync Members" flow adds members to the manually-created group

---

### Challenge 3 — Express Route Shadowing
**Problem:** The route `GET /:id` was catching requests to static paths like `/all` and `/project/:projectId` because they were registered after `/:id`. Express matched `id = "all"` first.

**Resolution:** Static paths (`/all`, `/create`, `/project/:projectId`) registered before any parameterized routes (`/:id`, `/:id/members`, etc.) in the route file. This is a critical ordering rule in Express.

---

### Challenge 4 — React Modal Showing Stale Member Data
**Problem:** After adding/removing a member, the `ManageGroupMembersModal` still showed the old member list because `managingGroup` state was set once on modal open and never updated.

**Resolution:** The `freshManagingGroup` pattern — reads the latest group data from the `groups` array by ID on every render. Since `addMember`/`removeMember` call `refresh()` which triggers a re-fetch, the modal is always showing current data without closing and reopening.

## 12. Notes for Future Reference

### Maytapi Phone Number Format
```
✅ Send to Maytapi createGroup/addGroupParticipant: "919617980134"  (digits only, no +)
✅ Store in ERP (members.phone):                    "+919617980134" (with +, E.164)
✅ URL param for DELETE member:                     "%2B919617980134" (encodeURIComponent)
```
`normalizePhone()` returns E.164 (`+91...`). Strip the `+` before sending to Maytapi.

### `tryProviderOp` — Always Use for External API Calls in Controllers
Any call to a WhatsApp provider should go through `tryProviderOp`. This keeps the hybrid model intact: ERP operations (MongoDB saves) always succeed regardless of provider state.

### Route Order in Express — Static Before Parameterized
```javascript
// ✅ CORRECT
router.get('/all', getAllGroups);           // static — matches first
router.get('/:id', getGroupById);          // parameterized — fallback

// ❌ WRONG
router.get('/:id', getGroupById);          // catches /all with id="all"
router.get('/all', getAllGroups);           // never reached
```

### `freshManagingGroup` Pattern — Keep Modals in Sync
When a modal renders data that can change (e.g., member list), don't store the data object in state. Store only the ID in state, then derive the current data from the live list:
```javascript
const freshManagingGroup = managingGroup
  ? (groups.find((g) => g._id === managingGroup._id) || managingGroup)
  : null;
```

### Kickstart Flag Auto-Patch
Creating a WhatsApp group via the ERP automatically marks the corresponding `kickstartData` flag on the Project. No manual step needed. The mapping is:
```
main → kickstartData.mainGroupCreated
drawing → kickstartData.drawingGroupCreated
supervision → kickstartData.supervisionGroupCreated
payment → kickstartData.paymentGroupCreated
```

### syncStatus State Machine
```
unsynced  ──[sync success]──→  synced
unsynced  ──[sync partial]──→  partial
unsynced  ──[sync failure]──→  failed
failed    ──[link manually]──→  unsynced  (via updateGroup resetting syncStatus)
unsynced  ──[re-sync success]──→  synced
partial   ──[re-sync success]──→  synced
```
