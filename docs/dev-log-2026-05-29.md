## 4. File Relationships & Data Flow

### Notification flow (end-to-end)

```
Backend event site (controller OR AI tool)
   ↓ calls
notificationDispatcher.dispatch({ type, module, title, link, recipients, actor, ... })
   ↓ resolves recipients
   - explicit IDs from caller
   - + cached admin IDs (60s TTL via getAdminUserIds())
   - − actor's own ID (unless notifyActor: true)
   ↓ dedupes via Set
   ↓ batch
Notification.insertMany(docs, { ordered: false })
   ↓ stored in MongoDB with indexes:
   { recipientId: 1, readAt: 1 }     // for unread-count
   { recipientId: 1, createdAt: -1 } // for latest list

Frontend (every 30s, paused when tab hidden):
NotificationContext.refresh()
   ↓ calls
notificationService.list({ limit: 10, skip: 0 })
   ↓ HTTP GET
/api/notifications?limit=10&skip=0
   ↓ controller
listNotifications(req, res)
   ↓ MongoDB
Notification.find({ recipientId: req.user.id }).sort({ createdAt: -1 })
   ↓ JSON { items, total, unread, limit, skip }
   ↓ Context state update
{ items, unreadCount }
   ↓ consumers re-render
NotificationBell  → badge text + dropdown
NotificationsPage → full inbox list
```

### Multi-attendee meeting flow

```
Schedule modal (MeetingsPage / MeetingsCalendarPage / LeadDetailsPage)
   ↓ user opens
   ↓ seedClientAttendeesForLead(lead) → initial client[] with relation 'lead'
   ↓ user adds internal staff via
MultiEmployeePicker (chip multi-select)
   ↓ user toggles notify checkboxes per attendee
AttendeesEditor
   ↓ on submit
crmService.createMeeting({ ..., attendees })
   ↓ HTTP POST /api/metting/create
createMeeting controller
   ↓ normalizeAttendees(req.body.attendees) validator
   ↓ Meeting.create({ ..., attendees: { internal, client } })
   ↓ notify({ type: 'meeting.scheduled', ... })
   ↓ on read
populate attendees.internal.userId so UI gets fresh data alongside snapshots
```

### Filter system flow

```
ClientsListPage / NewLeadsPage / etc.
   ↓ useFilters('crm', 'clients')
useFilters hook
   ↓ reads
FilterConfig.CRM_FILTERS.clients
   ↓ returns { filters, filterConfig, process, updateFilter, clearAllFilters }
   ↓ also syncs URL via setSearchParams
ClientsListPage renders
   ↓ <AdvancedFilter filters filterConfig updateFilter ...>
AdvancedFilter
   ↓ for each FILTER_TYPES key, conditionally renders
FilterDropdown (with multi-select checkbox UI) / SortSelector / DateRangeFilter
   ↓ user changes filter
updateFilter(key, value)
   ↓ filters state updates
   ↓ URL params update (replace: true)
   ↓ process(clients) re-runs
   = applyFilters(clients) → sortData
   ↓ paginated = filtered.slice(pageStart, pageEnd)
   ↓ render table rows
   ↓ Pagination footer
```

### DatePicker flow

```
EnquiryFormPage / ClientInfoFormPage
   ↓ <DatePicker name="dob" value={formData.dob} onChange={handleChange} ... />
DatePicker
   ↓ user clicks trigger button
   ↓ popover opens
   ↓ user changes Year dropdown → setViewMonth
   ↓ user changes Month dropdown → setViewMonth
   ↓ user clicks a day
   ↓ pick(date)
   ↓ clampDateToRange(date, minDate, maxDate)
   ↓ onChange({ target: { name: 'dob', value: '1985-02-14' } })
   ↓ form handleChange fires (existing function — no changes needed)
   ↓ setFormData((p) => ({ ...p, dob: '1985-02-14' }))
   ↓ on submit
backend POST /api/clients/create
```

## 5. Components Breakdown

### NotificationBell
- **Type:** Reusable (single instance currently in Navbar)
- **Purpose:** Top-bar bell icon with unread badge + dropdown of 10 most recent
- **Props:** none (reads NotificationContext)
- **Where used:** `Navbar.jsx`

### NotificationsPage
- **Type:** Page-level
- **Purpose:** Full inbox at `/notifications`
- **Props:** none (consumes context + service)
- **Inputs/State:** Module filter, unread-only toggle, currentPage (25/page)

### NotificationContext / NotificationProvider
- **Type:** Context provider (global state)
- **Purpose:** Single source of truth for notification state
- **Exposes:** `items`, `unreadCount`, `isLoading`, `lastFetchedAt`, `refresh`, `markAsRead`, `markAllAsRead`, `dismiss`
- **Where used:** wrapped at App root inside `ToastProvider`, outside `PMSProvider`

### DatePicker
- **Type:** Reusable input component
- **Purpose:** Drop-in replacement for native `<input type="date">` with month + year dropdowns for fast navigation
- **Props:** `label`, `name`, `value`, `onChange`, `icon`, `placeholder`, `min`, `max`, `yearRange`, `required`, `disabled`, `error`, `className`
- **Where used:** EnquiryFormPage (2x), ClientInfoFormPage (4x)

### MeetingCard
- **Type:** Reusable
- **Purpose:** Render a single meeting with all per-meeting actions
- **Props:** `meeting`, `onViewDetails`, `onStatusChange`, `onReschedule`, `onRecordMOM`
- **Where used:** MeetingsPage list, MeetingsCalendarPage day-meetings panel

### MeetingAttendeesSummary
- **Type:** Reusable sub-component (inside MeetingCard)
- **Purpose:** Collapsible `<details>` showing attendee names + notify-channel icons
- **Props:** `meeting`
- **Where used:** MeetingCard

### MeetingsCalendarPage
- **Type:** Page-level

- **Inputs/State:** Internal calendar state (currentMonth, selectedDate), modal state, fetched meetings + leads

### MultiEmployeePicker

## 8. State Management / Logic

### Notifications — frontend state
- **Global:** `NotificationContext` instance at App root
- **Local:** None — all consumers read from context
- **Polling:** 30s `setInterval`; halt when `document.visibilityState === 'hidden'`; refetch on `focus` + `visibilitychange`
- **Optimistic mutations:** state flips immediately on user action, falls back to `refresh()` on error
- **In-flight guard:** `inFlightRef` prevents overlapping fetches

### Notifications — backend logic
- **Best-effort:** all errors logged via `console.error`, never thrown
- **Admin cache:** 60s TTL on `getAdminUserIds()` — refreshed lazily on next dispatch
- **Recipient dedup:** stringifies ObjectIds for `Set` membership
- **Actor suppression:** drops actor's own ID from recipients by default
- **Empty-recipient short-circuit:** silent no-op if zero recipients

### Filter state (useFilters hook)
- **URL-synced:** `useSearchParams` with `replace: true` to avoid history pollution
- **Computed apiQuery:** ready for future server-side filtering
- **Two pipelines:** `applyFilters` (client-side) + `sortData` combined into `process()`
- **New filter types** (`lifecycleStage`, `source`) look up the model field path from filterConfig (supports nested paths like `leadId.lifecycleStage`)

### Pagination
- **Local `currentPage` state** per page
- **Reset to 1** via `useEffect([leads.length])` on filter changes
- **`safePage` clamps** to total pages so a shrinking list doesn't strand the user

### DatePicker internals
- **`viewMonth` state** — Date pointing at the 1st of the visible month
- **External `value` sync** via `useEffect([value])`
- **ISO parse helper** uses LOCAL Date components — never `new Date(str)` which interprets as UTC
- **`clampDateToRange`** enforces min/max
- **Edge alignment** computed via `getBoundingClientRect` on open

### Meetings calendar hover popover
- **Zero React state** — pure CSS `group/cell` + `group-hover/cell:` Tailwind classes
- **Edge-aware** via column-index + row-index math (Fri/Sat → align right; Sun → align left; last row → flip vertically up)
- **`pointer-events-none`** so popover doesn't block underlying cell hover/click

## 9. Challenges Faced

1. **Search input collapsed to 0px** in AdvancedFilter when many dropdowns competed for one row.
   - Cause: `flex-1 max-w-md` couldn't hold ground against intrinsic widths of 7 dropdowns.
   - Fix: restructured desktop layout — search on its own row by default; opt-in `inlineSearch={true}` for pages with few controls.

2. **Native `<input type="date">` UX bottleneck** — DOB fields required ~40 back-arrow clicks to reach 1985.
   - Diagnosed first as library bloat; confirmed via package.json audit there was none.
   - Fix: custom DatePicker with Year + Month dropdowns.

3. **App.jsx touched by both notification + calendar work** — couldn't cleanly split in two commits without `git add -p` (interactive add disallowed).
   - Fix: sequenced commits so App.jsx changes naturally land with notifications; commit message notes the calendar route bled in.

4. **AI tools bypass HTTP controllers** — user reported notifications not firing when they used the AI chatbot to create leads.
   - Cause: AI tools call models directly (`CRMClient.create(...)`), skipping the controller layer entirely.
   - Fix: audited all AI tool files, added `notify({...})` to each of 11 write tools with `notifyActor: true` + `metadata.viaAI: true` for distinguishing AI-originated events.

5. **Rescheduling wiped the "rescheduled" status** — discovered after user rescheduled but stat counter stayed at 0.
   - Cause: `handleRescheduleSubmit` on MeetingsPage was sending `status: 'scheduled'` (resetting), and `rescheduledFrom` was missing from both reschedule sites.
   - Fix: changed status to `'rescheduled'`, added `rescheduledFrom: selectedMeeting.date` on both MeetingsPage + LeadDetailsPage.

6. **Status badges wrapped mid-pill** — "Meeting Done", "Proposal Sent" broke into two lines because the column was too narrow.
   - Fix: added `whitespace-nowrap` + `inline-block` to the badge span.

7. **JSX parse error after large block deletion** in MeetingsPage — leftover stray `)}` from a removed ternary.
   - Caught via IDE diagnostic. Removed.

8. **FilterDropdown labels truncating** — "Meeting D..." instead of "Meeting Done".
   - Cause: dropdown menu inherited the trigger's narrow `w-32` via `left-0 right-0`.
   - Fix: `min-w-full w-max max-w-[260px]` so menu auto-fits content but never blows up.

9. **Tailwind JIT couldn't resolve dynamic class names** like `bg-${color}-500` in FilterDropdown colour dots.
   - Fix: switched to inline `style={{ backgroundColor: 'var(--{color})' }}` using CSS variables.

10. **Calendar felt cramped on the Meetings page** even after compaction (small calendar + meeting card side-by-side).
    - Fix: moved calendar to its own page `/crm/meetings/calendar` with full-width grid + day previews + hover popover.

11. **Native browser scrollbar arrows** showed inside the FilterDropdown menu — looked unprofessional.
    - Fix: applied the existing global `custom-scrollbar` utility.

12. **Multi-select dropdown affordance unclear** — users couldn't tell at a glance they could pick multiple Status values.
    - Fix: always-visible checkbox indicator on each row + "PICK ONE OR MORE" hint bar at the top of the dropdown.

## 12. Notes for Future Reference

### Architectural decisions made today
- **Best-effort dispatch:** notifications never block parent requests. If Mongo is unavailable, the user's lead-create still succeeds; only the notification is dropped (logged).
- **Admin fallback default:** every event copies admin/MD users unless `skipAdmins: true`. Keeps stakeholders in the loop without per-event configuration.
- **Polling over SSE for V1:** simpler, no infra, works behind proxies. SSE can be added later without changing emit sites.
- **Snapshot contact fields on Meeting.attendees:** future mail/WhatsApp dispatch should target the snapshotted phone/email, not re-resolve from the current User/CRMClient record (so historical notifications stay correct after later edits).
- **AI tools need their own `notify()` calls** — they bypass HTTP controllers entirely. Established this pattern today for 11 tools.

### Conventions to keep
- `type` field on Notification is `<module>.<verb>`, lowercase, dot-separated (e.g. `meeting.scheduled`, `proposal.sent`).
- `link` field is a frontend route path; the bell uses it to navigate on row-click.
- Bell colours: each module gets a colour-class map (CRM=amber, meeting=blue, pms=indigo, proposal=emerald, auth=purple, system=slate).
- DatePicker emits synthetic `{ target: { name, value } }` events for drop-in compatibility with existing `handleChange` handlers.
- Form `yearRange` is per-field scope:
  - DOB → 1920 to current year
  - Anniversary → 1960 to current year
  - Enquiry Date → 2020 to current+1
  - Preferred Meeting Date → current to current+2
- `inlineSearch={true}` on AdvancedFilter when ≤4 dropdowns; leave default (stacked) when ≥6.

### Backend gotchas
- The Meeting model file is `Metting.model.js` (typo — preserve until anyone bulk-renames).
- `req.user` shape is `{ id, email, role }` — `id` (not `_id`).
- AI tool `ctx.userId` is a string; coerce to ObjectId before model writes.

### Frontend gotchas
- Tailwind dynamic class names like `bg-${color}-500` are NOT resolved by JIT. Use inline `style` with CSS variables instead.
- `<details>` summary list-marker styling is browser-specific. Use `list-none` on the summary.
- `pointer-events-none` on a popover lets the cell underneath stay hoverable but disables scrolling the popover content.
- Date strings: always parse `YYYY-MM-DD` manually (use local Date components); never `new Date(str)` which interprets as UTC.
